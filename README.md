# Spectral Bounce

Bouncing shapes whose colours combine as **real spectral filters**, not RGB
blends. Circles, freely-rotating squares/rectangles, and freely-rotating font
glyphs drift around with elastic collision physics and overlap. Where they
cross, their colours combine the way physical media actually combine light:

- **Subtractive mode** (default): coloured **filters** on a white background.
  Overlapping shapes *multiply* their transmittance spectra, exactly like
  stacking gels — red over cyan goes nearly black, yellow over cyan goes green,
  and so on. The white background is **6500 K blackbody** light.
- **Additive mode**: the shapes are **luminous** on a black background and their
  emitted spectra *add* — overlaps brighten toward white.

The colour of every overlap is computed in **spectral space** (16 wavelength
bands, 400–700 nm) and only converted to sRGB at the very end, by integrating
against the CIE 1931 colour-matching functions. This is done on the GPU with a
WebGL2 multiple-render-target pipeline so it runs per-pixel in real time.

**Live demo:** <https://inhahe.com/spectral-bounce/>

## Run it

Because the app loads spectral data files, the most reliable way is a tiny local
server:

```
cd "spectral-bounce"
python -m http.server 8000
```

then open <http://localhost:8000/> .

It also works by simply double-clicking `index.html` (opening from `file://`),
because the spectral data is additionally embedded in `spectra_bundle.js` as a
fallback.

## Controls

- **Mode** — subtractive (filters/white) or additive (luminous/black).
- **Shape classes** — enable any mix of circles, squares, rectangles, glyphs.
  Shapes pass freely *through* one another (so their colours overlap and
  combine) and only bounce off the walls.
- **Glyphs**
  - **Font** — a drop-down of common/system fonts, plus **Load font file** to
    load an arbitrary `.ttf/.otf/.woff/.woff2` from disk.
  - **Character set** — *Unicode* (the whole printable range, sampled at random —
    the default), *Emoji* (rendered as filter-tinted silhouettes, since only the
    glyph's coverage feeds the spectral compositing), *ASCII*
    (letters/numbers/symbols &lt; 127), *CP437* (the classic DOS code page), or
    *Custom* (type your own string). Characters the chosen font can't actually
    draw are skipped, so you never get “tofu” boxes.
- **Population** — number of shapes; **Speed (mean)** and **Speed spread (σ)**:
  each shape's speed is drawn from a normal (bell-curve) distribution whose mean
  and standard deviation you set live; and **Brightness**. In *subtractive* mode
  brightness changes the **filters only** — the white background (or the
  illuminant's tint) stays fixed while the coloured shapes darken toward black or
  brighten toward the background across the slider's full range. In *additive*
  mode it is an ordinary linear exposure on the black background.
- **White light** — choose the illuminant SPD (default 6500 K blackbody). The
  6500 K reference reads as neutral white; other lights visibly tint the whole
  scene (warm under 3200 K, cool under 9500 K) and shift every filter's colour.
- **Filters in play** — toggle which filter envelopes get randomly assigned.
- **Respawn / Pause**.

### Deep-link / URL parameters

The initial configuration can be set from the query string (handy for sharing a
particular look), e.g.:

```
?mode=additive&classes=circle,glyph&count=30&charset=unicode&font=Georgia&illuminant=Blackbody%203200K&brightness=140&spread=60
```

Supported keys: `mode` (`subtractive`|`additive`), `classes` (comma list of
`circle,square,rect,glyph`), `count`, `speed`, `spread` (bell-curve σ, 0–150),
`brightness` (global output multiplier, 0–400; `exposure` accepted as an alias),
`font`, `charset` (`unicode`|`emoji`|`ascii`|`cp437`|`custom`), `chars` (a
literal string; implies `charset=custom`), `illuminant`, and `nofloat` (force
the 8-bit spectral buffer — see below).

## Spectral data files (`spectra/`)

All data is generated by `gen_spectra.py` on a 380–730 nm, 5 nm grid:

| File | Contents |
|------|----------|
| `cmf.json` | CIE 1931 2° colour-matching functions (x̄, ȳ, z̄) |
| `illuminants.json` | White-light SPDs — 6500 K blackbody (default) + others |
| `filters.json` | Transmittance envelopes for ~18 standard-ish filters |

To edit or add spectra, change `gen_spectra.py` and re-run it:

```
python gen_spectra.py
```

This rewrites the JSON files **and** the `spectra_bundle.js` fallback.

## How the colour maths works

1. Each shape carries a 16-band spectrum (a filter transmittance `T(λ)`).
2. Shapes are rasterised into a float MRT framebuffer holding those 16 bands
   across 4 RGBA textures:
   - subtractive: buffer starts at 1.0, each shape **multiply-blends** `T(λ)`;
   - additive: buffer starts at 0.0, each shape **add-blends** `white(λ)·T(λ)`.
3. A resolve pass reconstructs the 16-band spectrum per pixel, multiplies by the
   white SPD (subtractive only), integrates against the CMFs to get CIE XYZ,
   converts XYZ→linear sRGB (D65), applies a **fixed** white balance keyed to the
   *default* illuminant (so that light reads as neutral white while other lights
   tint the scene warm/cool rather than being normalised away), applies the
   brightness control, and gamma-encodes. Brightness is a linear exposure in
   additive mode; in subtractive mode it is a per-channel power curve anchored at
   the background colour (`bg·(rgb/bg)^(1/brightness)`), so the background is left
   untouched — `pow(1)==1` — while only the filters, which sit below the
   background, change brightness.

## Requirements

A browser with **WebGL2** (all current Chrome/Edge/Firefox). Floating-point
render targets (`EXT_color_buffer_float`) are used when available for full
spectral precision; otherwise it falls back to an 8-bit buffer. Real GPUs and
current software rasterizers (SwiftShader) both use the float path normally. As
a safety net, at startup the app resolves one empty subtractive frame into a
small offscreen buffer and checks it reads back as white; a driver that reports
the float extension but renders the target as black fails that check and is
dropped to the universally-supported 8-bit buffer. The `?nofloat` URL override
forces the 8-bit buffer explicitly.
